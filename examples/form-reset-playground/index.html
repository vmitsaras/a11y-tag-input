Form behavior lab

Track Values Through Reset

A QA engineer can see exactly which value is authored, edited, submitted, and restored.

Scenario

User goal: verify that visible tags, the synchronized form control, submission, and native reset stay aligned.

Try it

  • Add or remove tags and compare Current with Initial.
  • Submit a snapshot, then continue editing.
  • Reset the form and confirm Current and Reset restore the defaults.
  • Notice that reset leaves focus on the reset button and stays silent.

How to build it

Listen for the render event to show current state. On reset, wait one microtask for the plugin's native reset reconciliation before reading getTags().

form.addEventListener("reset", () => {
  queueMicrotask(() => updateReset(instance.getTags()));
});

Accessibility notes

  • The comparison is a description list with visible labels.
  • Changing Current is intentionally non-live, avoiding announcements on every edit.
  • The native reset remains silent and does not move focus.
  • JSON array formatting makes empty and multilingual values explicit.

Limitations

The page intercepts submission only to display a snapshot. Production forms must provide their own success, error, and recovery behavior.
