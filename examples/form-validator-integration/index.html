Real-world integration / Editorial guide

Publish With One Clear Error Path

Editors need useful topic tags and a form that explains every blocked submission. This example lets two accessibility plugins collaborate without competing for the tag field's error state.

Scenario

User goal: prepare an article with a title, editor contact, and at least two committed topic tags before publishing.

Try it

  1. Submit the empty form and use the focused summary links.
  2. Enter not-an-email, type draft without pressing Enter, and submit again.
  3. Commit accessibility with Enter and confirm one tag is still insufficient.
  4. Add forms, correct the other fields, and publish successfully.
  5. Use Reset draft to return to the initial state without stale errors.

One owner per error

A11y Form Validator checks the whole form and builds the summary. For the generated tag field, its adapter calls setValidationError() instead of creating a second inline error. 

const tagInput = createTagInput(source);

const validator = createFormValidator(form, {
  ...createDefaultPreset(),
  validateOn: ["submit"],
  renderer: createTagRenderer(tagInput),
  rules: {
    [tagInput.field.id]: {
      "minimum-tags": { min: 2 }
    }
  }
});

How the integration works

Initialization order

  1. Enhance the semantic textarea with A11y Tag Input.
  2. Create a renderer that recognizes the generated text field.
  3. Initialize A11y Form Validator so it discovers that field.
  4. Register the custom rule before the user can submit.

Value boundary

The form rule reads tagInput.getTags(). Text still sitting in the generated input is a draft, so it cannot accidentally satisfy the minimum-tag requirement.

Accessibility notes

  • Labels and hints exist before either plugin initializes.
  • Blocked submission focuses one error summary with field-labelled links.
  • A11y Tag Input remains the only owner of the tag field's visible error, aria-invalid, and aria-errormessage.
  • Candidate-tag feedback uses the tag input status region; form-level tag-count feedback uses the summary without moving focus during recovery.
  • Visible error text and summary links avoid color-only feedback.
  • Focus motion respects reduced-motion preferences, and controls retain forced-colors boundaries.

Limitations

  • The adapter maps one generated tag field to one tag input instance; repeat the mapping deliberately for multiple widgets.
  • Tag changes refresh form-level state after the first submit, but only a blocked submit moves focus to the summary.
  • Client-side validation improves recovery but does not replace server-side validation.
  • Test the final integration with the browsers, screen readers, zoom levels, touch input, and language settings your product supports.
