Validation Hints For Automation Slugs

Operations teams need compact automation tags that work in URLs, logs, and filters. This recipe explains the rules before validation fails.

Scenario

User goal: create machine-friendly tags using lowercase letters, numbers, and hyphens.

Try it

  • Add build-queue.
  • Try QA and inspect the lowercase transform.
  • Try bad tag to trigger the invalid character message.
  • Try a one-character tag to trigger the minimum length message.

Example

Use 2 to 20 characters. Lowercase letters, numbers, and hyphens only.

Developer notes

The core options cover common validation rules. Use custom validators for rules that need domain logic. 

createTagInput(source, {
  lowercase: true,
  minLength: 2,
  maxLength: 20,
  invalidChars: /[^a-z0-9-]/
});

How to build it

Describe the rules beside the field, connect that hint with aria-describedby, and configure the same constraints in JavaScript.

HTML

<p id="automation-tags-help">
  Use 2 to 20 characters. Letters, numbers, and hyphens only.
</p>
<textarea id="automation-tags"
  aria-describedby="automation-tags-help"></textarea>

JavaScript

createTagInput(source, {
  lowercase: true,
  minLength: 2,
  maxLength: 20,
  invalidChars: /[^a-z0-9-]/
});

Accessibility notes

  • The rule summary is visible before validation runs.
  • Rejected tags use the plugin's visible error and live status message.
  • The example avoids color-only validation feedback.
  • Use clear recovery text for custom validators in production forms.

Limitations

  • This recipe demonstrates addon-style behavior around the core plugin; production forms still need target browser, assistive technology, touch, zoom, and forced-colors checks.
  • Keep labels, validation copy, and recovery instructions aligned with your own domain language before shipping.
