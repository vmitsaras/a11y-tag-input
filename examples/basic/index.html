A11y Tag Input

A static package demo for progressively enhancing a native textarea into a keyboard-friendly tag input.

Scenario

A developer is evaluating the core tag input behavior before deciding which addon-style recipe matches a production form.

Quick Facts

No framework Imports the built ESM package.

Native form value Keeps the source textarea synchronized.

Keyboard support Enter, separators, arrows, Home, End, Backspace, Delete.

CSS hooks Uses scoped custom properties and state classes.

What To Try

  1. Type Design and press Enter. The tag becomes design.
  2. Type Research;Forms to add multiple tags with a separator.
  3. Press Backspace from an empty input to move focus to the final remove button.
  4. Use ArrowLeft, ArrowRight, Home, and End to move through remove buttons and back to the input.
  5. Press Tab from a remove button to leave the tag editor without visiting every tag.
  6. Press Delete or Backspace on a focused remove button.
  7. Try adding duplicate tags or more than five tags to inspect validation feedback and recovery wording.

Installation And Usage

Build the package, import its JavaScript and stylesheet, then initialize the semantic textarea shown in the HTML snippet below.

Build locally

npm install
npm run build

The npm registry package was not verified during this docs pass, so use a local checkout or packed tarball until publication is confirmed.

JavaScript

import { createTagInput } from "a11y-tag-input";
import "a11y-tag-input/styles.css";

const source = document.querySelector("[data-a11y-tag-input]");

if (source instanceof HTMLTextAreaElement) {
  createTagInput(source);
}

HTML

<div class="a11y-tag-input" data-a11y-tag-input-root>
  <label for="example-tags">Topics</label>
  <textarea
    id="example-tags"
    name="topics"
    data-a11y-tag-input
    data-max-tags="5"
    data-lowercase="true"
  >accessibility,forms</textarea>
</div>

Accessibility Behavior To Inspect

  • The generated input keeps the visible label and field description.
  • Selected tags render as a labelled list with real remove buttons.
  • Add, remove, duplicate, paste, and limit messages update a polite status region.
  • Validation errors use aria-invalid, aria-errormessage, and visible error text.
  • Default CSS includes visible focus, forced-colors support, and reduced-motion handling.

This demo shows implemented behavior. It is not a substitute for testing your production form with target assistive technologies.

Limitations

  • Screen-reader speech, physical touch keyboards, browser zoom, and Windows High Contrast still require real-device/manual checks.
  • The generated input enhances a native control; verify serialization, labels, and validation in the final production form.
