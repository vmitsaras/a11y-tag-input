Duplicate Feedback For Release Labels

Release managers need duplicate labels rejected with clear, calm feedback instead of silent failure.

Scenario

User goal: build a small release label set and understand immediately when a duplicate tag is rejected.

Try it

  • Type docs and press Enter.
  • Type security and press Enter.
  • Type docs again and inspect the duplicate feedback.
  • Remove a tag and add it again.

Example

Add unique labels for the release note filter.

Developer notes

Duplicate rejection is built into the core plugin when allowDuplicates is false. 

createTagInput(source, {
  allowDuplicates: false,
  messages: {
    duplicateTag: "{tag} is already in this release."
  }
});

How to build it

Start with a semantic textarea, then customize the duplicate message while the plugin handles visible and announced feedback.

HTML

<textarea id="release-labels" name="release-labels">
docs,qa
</textarea>

JavaScript

const source = document.querySelector("#release-labels");

createTagInput(source, {
  allowDuplicates: false,
  messages: {
    duplicateTag: "{tag} is already in this release."
  }
});

Accessibility notes

  • The plugin sets aria-invalid and aria-errormessage for rejected duplicates.
  • The plugin exposes one canonical field-level error for rejected duplicates.
  • The separate demo output is reserved for successful actions.
  • Keyboard users stay in the field after a duplicate rejection so they can edit the value.
  • Test final copy with target assistive technologies before shipping.

Limitations

  • This recipe demonstrates addon-style behavior around the core plugin; production forms still need target browser, assistive technology, touch, zoom, and forced-colors checks.
  • Keep labels, validation copy, and recovery instructions aligned with your own domain language before shipping.
